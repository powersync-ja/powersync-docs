Data doesn't always flow in the direction of your firewall rules, so the below table documents which components are making connections to others:

| Request Originator | Request Destination        | Protocol    |
| ------------------ | -------------------------- | ----------- |
| PowerSync Service  | Postgres                   | TCP         |
| PowerSync Service  | MongoDB                    | TCP         |
| PowerSync Service  | OpenTelemetry Collector    | TCP or UDP  |
| PowerSync Service  | JWKS Endpoint              | TCP (HTTPS) |
| App Client         | PowerSync Service (via LB) | TCP (HTTPS) |
| App Client         | App Backend                | TCP (HTTPS) |
| App Backend        | Postgres                   | TCP         |

### JWKS Endpoint Reachability

If you are using a JWKS URI for client authentication (see [Client Authentication](/configuration/auth/overview)), the PowerSync Service must be able to reach your JWKS endpoint over outbound HTTPS. Ensure that:

- Your firewall or network policy permits outbound HTTPS (port 443) from all PowerSync containers to your auth service.
- Your JWKS endpoint is reachable from the network where PowerSync runs. If PowerSync is in a private VPC, your auth service must be accessible from that VPC (e.g. via a private endpoint or an allowlisted egress rule).
- If your auth service uses IPv6 addresses, verify that your PowerSync environment also supports IPv6. IPv6-only JWKS endpoints can cause connection failures. See [Error Codes](/debugging/error-codes) for related errors.

If you are not using a JWKS URI (for example, if you use inline public keys or a custom auth implementation), this requirement does not apply. See [Custom Authentication](/configuration/auth/custom) for alternatives.
